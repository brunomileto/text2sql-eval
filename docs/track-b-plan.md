# Track B Implementation Plan

## Summary

Implement Track B as "question + schema metadata enrichment" using the existing
track and prompt architecture.

Track B will:

- introspect the active SQLite database once at the start of each run
- keep that schema in memory for all records in the run
- save the structured schema once per run as `results/{run_id}/schema_context.json`
- reference that file from `run.json` in `run_metadata`
- render prompt-ready schema markdown in memory and inject it into
  `config/prompts/track_b.txt`

Artifact version stays `v1`.

## Implementation Changes

### Schema extraction and reuse

- Replace the stub in `parse_schema(...)` with real SQLite introspection.
- Build `SchemaContext` from the active DB using:
  - table discovery from SQLite catalog, excluding internal `sqlite_%` tables
  - column/type/PK from `PRAGMA table_info`
  - FK/reference info from `PRAGMA foreign_key_list`
- Keep descriptions empty for v1; source of truth is SQLite only.
- Preserve the current run-scoped lifecycle in `load_questions(...)`: parse
  schema once, attach the same `SchemaContext` to every `EvalQuestion`.
- Ensure the schema is available even if a future track other than B needs it.

### Track B behavior

- Add `TrackB` following the same shape as `TrackA`.
- Register `"b"` in the track registry.
- `TrackB.build_prompt(...)` should call `render_prompt("track_b", {...})`.
- Track B prompt context must include:
  - `question`
  - `schema`
- `schema` is the markdown generated by `SchemaContext.to_markdown()`.
- `TrackB.pre_fetch(...)` remains the default no-op.
- `TrackB.build_artifacts(...)` should stay minimal for v1; do not duplicate the
  schema payload in each record. The prompt plus run-level schema artifact is the
  source of truth.

### Prompt and artifact contract

- Add `config/prompts/track_b.txt` using the same external prompt pattern as
  Track A.
- Required placeholders for Track B:
  - `{question}`
  - `{schema}`
- Extend `RunMetadata` with one additive field:
  - `schema_artifact_path: str | None`
- Keep `schema_version` as `v1`.
- Reporter should write `schema_context.json` into the run folder before writing
  `run.json`.
- `schema_artifact_path` should be stored as a run-relative filename, not an
  absolute path:
  - `schema_context.json`
- Save only structured JSON to disk.
- Do not write a markdown schema file; markdown is generated in memory when
  building Track B prompts.

### Docs and contract updates

- Update prompt docs to describe Track B placeholders and file naming.
- Update artifact docs to document the new
  `run_metadata.schema_artifact_path` field and the new sibling file
  `schema_context.json`.
- Update README examples to reflect that Track B now exists and uses schema
  injection.

## Public Interfaces and Contract Changes

- Track registry now supports `b` as an implemented track.
- `run.json` `run_metadata` gains:
  - `schema_artifact_path: str | None`
- New sibling artifact in each run directory:
  - `schema_context.json`
- New prompt template:
  - `config/prompts/track_b.txt`

## Test Plan

- Schema parser unit tests:
  - parses tables and columns from a temporary SQLite DB
  - captures PK flags
  - captures FK flags and references
  - ignores internal SQLite tables
  - returns empty `SchemaContext` for DBs with no user tables
- Track tests:
  - `get_track("b")` returns `TrackB`
  - Track B prompt includes the question and rendered schema markdown
  - Track B differs from Track A by using schema context
- Prompt tests:
  - `track_b.txt` loads successfully
  - rendering fails clearly if `{question}` or `{schema}` is missing
- Pipeline and reporter tests:
  - `run.json` includes `run_metadata.schema_artifact_path`
  - `schema_context.json` is created in the run folder
  - saved JSON matches the introspected schema structure
- End-to-end integration test:
  - local SQLite fixture with FK relationship
  - run with fake provider and `tracks: [b]`
  - verify prompt contains schema markdown
  - verify `schema_context.json` exists and is referenced from `run.json`

## Assumptions and Defaults

- Track B v1 uses SQLite introspection only; no YAML-enriched descriptions.
- Schema is computed once per run and reused across all records in that run.
- The separate schema artifact is run-scoped, not track-scoped or record-scoped.
- The artifact reference belongs in `run_metadata`, not in each record.
- The on-disk schema artifact is JSON only; markdown remains in-memory for prompt
  rendering.
- Output contract remains `v1` because the new metadata field is additive.
